# Architecture

Высокоуровневая схема Personal Financial Tracker. Здесь — про модули, БД и решения,
почему они выбраны именно так.

## Сообщение → запись: пайплайн

1. `bot/middleware/rateLimit.ts` — Redis INCR + TTL: ≤ 30 апдейтов в минуту на user_id.
2. `bot/handlers/text.ts` — сначала проверяем «pending intent» (например, ввод суммы
   цели или бюджета) — если есть, обрабатываем и выходим.
3. `parser.ts` — отправляем текст в Groq в JSON-mode с заточенным system-промптом.
   Парсер умеет извлекать массив транзакций (`такси 200, кофе 150, обед 650`).
4. `categories.ts::normalizeCategory()` — нормализуем категорию по словарю синонимов.
5. `transactions.ts::insertTransaction()` — пишем в Postgres (`amount` всегда в копейках,
   `bigint`).
6. `undo.ts::storeUndo()` — кладём токен в Redis на 5 минут, чтобы пользователь мог
   отменить запись inline-кнопкой.
7. `budgetCheck.ts` — после записи смотрим, перевалила ли категория за 80% и 100%
   от месячного лимита.

## Деньги — только bigint

Все суммы хранятся и считаются в копейках (`bigint` в Postgres, `number ≤ Number.MAX_SAFE_INTEGER`
в Node). На дисплей конвертируется через `format.ts::formatAmount()`. Это исключает
классический `0.1 + 0.2 !== 0.3` и проблемы округления.

## Время и таймзоны

`src/time.ts` — единственный модуль, знающий про IANA-зоны. Везде остальное оперирует
`Date` в UTC. Начало дня/недели/месяца/квартала/года считается через `Intl.DateTimeFormat`
с указанной зоной (`Europe/Moscow` по умолчанию) — этого достаточно для жителей РФ/СНГ
и не требует подкаченных tz-данных.

## Charts

`chart.ts` использует `chartjs-node-canvas` + системные шрифты `DejaVu Sans` /
`Noto Color Emoji`. На сервере отрисовывается:

- **Donut** — недельный отчёт по категориям, с подписями процентов и сумм.
- **Bar** — дневная динамика недели (расход + доход), фон `#15171c`.

PNG отдаётся в Telegram как `photo`.

## Подписки

`subscriptions` хранит шаблон (`amount`, `category`, `cadence`, `next_charge`),
а cron каждые 5 минут вызывает `chargeDueSubscriptions()`: если `next_charge <= now`,
вставляется фактическая транзакция, `next_charge` смещается на одну `cadence`-итерацию,
и пользователь получает уведомление.

Если бот лежал, пайплайн идемпотентен: при следующем тике пробежимся по всем
просрочкам подряд (до 50 итераций за раз — sanity-cap).

## Цели

`goals` — простая таблица `(user_id, month, target_amt)`. Сравниваем `getRangeTotals()`
за текущий месяц с `target_amt`. Цели только на текущий месяц (без истории — KISS).

## Redis: что и зачем

- `undo:<token>` — TTL 5 мин, маппит токен на массив `transaction.id`.
- `pending:<user>` — JSON-объект текущего намерения (custom-budget, goal, search, sub flow).
- `rl:<user>:<window>` — счётчик per-user rate-limit.

## Schedules

| Cron          | Тайм-зона       | Что                                |
| ------------- | --------------- | ---------------------------------- |
| `0 21 * * *`  | tz пользователя | напоминание «не забыл записать?»   |
| `0 20 * * 0`  | tz пользователя | недельный отчёт + donut + AI-совет |
| `*/5 * * * *` | UTC             | списания по подпискам              |

## Observability

- `GET /healthz` — `200 ok` если процесс жив.
- `GET /readyz` — `200 ok` только если Postgres и Redis отвечают.
- Логи структурированы (`[ISO] [LEVEL] message`), уровень управляется `LOG_LEVEL`.

## Тестирование

- `vitest` — unit-тесты на чистые функции (категории, формат, время, подписки).
- Smoke-test через `npm run dev` с dummy-токеном — стартует процесс, прогоняет
  миграции, подключается к Postgres+Redis. Используется как локальный sanity-check.
